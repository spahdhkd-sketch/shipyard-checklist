# 불안전요소 등록 및 호선자재 누락 상세화면 설계

## 목표

홈 화면의 핵심 메뉴에 추가되는 `불안전요소 등록`과 `호선자재 누락`을 실제 업무 흐름에 맞는 상세 기능으로 설계한다.

두 기능은 작업자가 현장에서 빠르게 등록할 수 있어야 하며, 관리자는 별도 관리 화면에서 작업자 목록과 등록 기록의 처리 상태를 관리할 수 있어야 한다.

## 범위

- `불안전요소 등록` 작업자 등록 화면
- `호선자재 누락` 작업자 등록 화면
- 등록 완료 후 상세/완료 화면
- 관리자 모드에서만 보이는 `관리` 메뉴
- `관리` 화면의 탭 구조
- 작업자 목록 관리
- 불안전요소 및 자재누락 기록 목록/상세/상태 관리
- Supabase 테이블 및 Storage 저장 방향
- 향후 처리 이력 타임라인 확장 메모

홈 화면의 카드 추가 자체는 별도 작업으로 진행한다. 이 문서는 카드 클릭 이후의 상세 기능과 데이터 구조를 정의한다.

## 공통 원칙

- 작업자 화면은 빠른 입력을 우선한다.
- 관리자 화면은 상태 추적과 정리 업무를 우선한다.
- 등록자의 이름은 직접 입력하지 않고 관리자 관리 작업자 목록에서 선택한다.
- 두 등록 화면의 `호선` 선택지는 현재 작업자 점검 화면에 공개된 호선만 사용한다.
- 현재 앱 기준으로 작업자에게 공개되는 호선은 L/C일이 입력된 호선이다.
- 모든 기록은 `등록일시`, `수정일시`, `완료일시` 정보를 가진다.
- 관리자만 기록을 삭제할 수 있으며 삭제는 영구 삭제다.
- 관리자 상세에서는 작업자가 입력한 원본 값은 수정하지 않고, 상태와 조치/메모만 수정한다.

## 불안전요소 등록

### 성격

작업자가 현장에서 불안전 상태나 위험 요소를 빠르게 등록하는 화면이다. 관리자는 이후 `관리` 화면에서 접수된 기록을 확인하고 조치 상태를 변경한다.

### 작업자 입력 필드

- `호선`
- `내용`
- `사진`
- `등록자`

### 제외 필드

초기 버전에서는 다음 필드를 두지 않는다.

- 위치
- 위험 유형
- 긴급도
- 조치 담당자

### 사진 정책

- 사진은 권장이다.
- 사진 없이 등록하려 하면 확인창을 표시하고, 사용자가 확인하면 등록을 허용한다.
- 한 기록당 사진은 최대 3장까지 첨부할 수 있다.
- 사진은 Supabase Storage에 저장한다.
- DB에는 사진 파일 자체가 아니라 Storage path 목록을 저장한다.
- 목록에서는 대표 사진 1장을 썸네일로 표시한다.
- 상세 화면에서는 최대 3장을 모두 표시한다.

권장 Storage 구조:

```text
bucket: issue-photos

unsafe/
  {recordId}/original-1.jpg
  {recordId}/original-2.jpg
  {recordId}/original-3.jpg
```

### 상태 흐름

`접수 → 조치중 → 완료`

등록 직후 기본 상태는 `접수`다. 관리자가 상태를 변경하면 `수정일시`를 갱신한다. 상태가 `완료`가 되면 `완료일시`를 저장한다.

### 목록 구조

관리 화면의 `불안전요소` 탭에서는 상태별 그룹으로 표시한다.

- `접수`: 기본 펼침
- `조치중`: 기본 펼침
- `완료`: 기본 접힘

각 그룹 안에서는 최신 등록순으로 정렬한다.

상단 필터:

- 호선
- 상태
- 등록자

상단 정렬:

- 상태 우선순
- 최신 등록순
- 호선 번호순
- 등록자순

상태별 그룹의 단점은 같은 호선의 이슈가 여러 상태 그룹에 흩어질 수 있다는 점이다. 이는 호선 필터로 보완한다. 완료 기록이 많아지는 문제는 완료 그룹 기본 접힘과 상태 필터로 보완한다.

## 호선자재 누락

### 성격

작업자가 특정 호선의 누락 자재를 빠르게 등록하고, 담당자/관리자가 호선별 누락 현황을 관리하는 화면이다.

### 작업자 입력 필드

- `호선`
- `자재명`
- `내용`
- `등록자`

초기 버전에서는 수량과 규격을 필수 입력으로 두지 않는다. 필요한 세부 정보는 `내용`에 작성한다.

### 상태 흐름

`접수 → 확인중 → 완료`

등록 직후 기본 상태는 `접수`다. 관리자가 상태를 변경하면 `수정일시`를 갱신한다. 상태가 `완료`가 되면 `완료일시`를 저장한다.

### 목록 구조

관리 화면의 `자재누락` 탭에서는 기본적으로 호선별 그룹으로 표시한다.

- 전체 보기에서는 등록 기록을 호선별로 묶어 표시한다.
- 상단 호선 필터에서 특정 호선을 선택하면 해당 호선 기록만 표시한다.
- 각 호선 그룹 안에서는 미완료 기록을 우선 표시한다.
- 완료 기록은 기본 접힘으로 표시한다.

상단 필터:

- 상태
- 호선
- 등록자
- 자재명

상단 정렬:

- 상태 우선순
- 최신 등록순
- 호선 번호순
- 등록자순
- 자재명순

## 등록 완료 화면

두 등록 화면 모두 등록이 완료되면 홈으로 바로 이동하지 않고, 방금 등록한 기록의 상세/완료 화면을 표시한다.

### 불안전요소 등록 완료 화면

표시 정보:

- 호선
- 내용
- 등록자
- 사진 첨부 여부 및 첨부 사진
- 등록일시
- 현재 상태: `접수`

### 호선자재 누락 등록 완료 화면

표시 정보:

- 호선
- 자재명
- 내용
- 등록자
- 등록일시
- 현재 상태: `접수`

### 완료 화면 버튼

- `추가 등록`: 가장 큰 기본 버튼
- `목록 보기`: 보조 버튼
- `홈으로`: 보조 버튼

## 관리 메뉴

### 접근 방식

`관리` 메뉴는 관리자 모드가 ON일 때만 사이드/하단 메뉴에 표시한다. 작업자 모드에서는 숨겨 작업자 화면을 단순하게 유지한다.

현재 관리자 모드 진입 방식은 기존 앱의 비밀번호 방식인 `gs2026`을 따른다.

### 내부 구조

`관리` 화면은 탭 3개로 구성한다.

- `작업자`
- `불안전요소`
- `자재누락`

탭 구조를 사용해 모바일에서도 화면을 안정적으로 유지하고, 관리 기능이 늘어나도 한 화면이 과도하게 길어지지 않게 한다.

## 작업자 목록 관리

### 작업자 정보

- 이름
- 소속/팀

### 관리자 기능

- 작업자 추가
- 작업자 수정
- 작업자 삭제

작업자 삭제 시 해당 작업자는 이후 등록자 드롭다운에 표시되지 않는다. 다만 기존 기록에는 등록 당시의 작업자 이름과 소속/팀 스냅샷을 그대로 남긴다.

기록에는 가능하면 다음 값을 함께 저장한다.

- `workerId`
- `workerNameSnapshot`
- `workerTeamSnapshot`

이렇게 하면 작업자 목록에서 이름이나 팀이 바뀌거나 삭제되어도 과거 기록의 책임 추적이 유지된다.

## 관리자 상세 및 수정 정책

관리자는 기록 상세에서 다음 값만 수정할 수 있다.

- 상태
- 조치/메모

다음 원본 입력값은 수정하지 않는다.

- 호선
- 내용
- 자재명
- 등록자
- 등록 사진

초기 버전에서 조치/메모는 기록당 1개만 둔다. 상태와 조치/메모를 수정하면 `수정일시`를 갱신한다. 상태가 완료가 되면 `완료일시`를 저장한다.

## 삭제 정책

- 관리자만 삭제할 수 있다.
- 삭제는 영구 삭제다.
- 삭제 전 확인창을 표시한다.
- 불안전요소 기록 삭제 시 연결된 Supabase Storage 사진도 함께 삭제한다.
- 호선자재 누락 기록 삭제 시 해당 DB 기록을 삭제한다.

## Supabase 저장 구조

초기 버전에서는 다음 4개 저장 단위를 사용한다.

- `workers`
- `unsafe_issues`
- `missing_materials`
- `issue_photos`

### workers

작업자 드롭다운에 표시할 이름과 소속/팀을 관리한다.

주요 필드:

- `id`
- `name`
- `team`
- `created_at`
- `updated_at`

### unsafe_issues

불안전요소 기록을 저장한다.

주요 필드:

- `id`
- `ship_no`
- `content`
- `worker_id`
- `worker_name_snapshot`
- `worker_team_snapshot`
- `status`
- `admin_memo`
- `created_at`
- `updated_at`
- `completed_at`

상태 값:

- `접수`
- `조치중`
- `완료`

### missing_materials

호선자재 누락 기록을 저장한다.

주요 필드:

- `id`
- `ship_no`
- `material_name`
- `content`
- `worker_id`
- `worker_name_snapshot`
- `worker_team_snapshot`
- `status`
- `admin_memo`
- `created_at`
- `updated_at`
- `completed_at`

상태 값:

- `접수`
- `확인중`
- `완료`

### issue_photos

불안전요소 사진의 Storage path를 저장한다.

주요 필드:

- `id`
- `target_type`
- `target_id`
- `storage_bucket`
- `storage_path`
- `sort_order`
- `created_at`

초기 버전에서는 `target_type`이 `unsafe_issue`인 사진만 사용한다. 향후 호선자재 누락에도 사진이 필요해지면 같은 테이블과 Storage 구조를 재사용할 수 있다.

## Supabase Storage

사진은 Supabase Storage bucket에 저장한다.

권장 bucket:

```text
issue-photos
```

권장 path:

```text
unsafe/{recordId}/original-1.jpg
unsafe/{recordId}/original-2.jpg
unsafe/{recordId}/original-3.jpg
```

Storage 정책은 현재 프로젝트가 승인한 `anon` 쓰기 허용 구조와 호환되어야 한다. 구현 단계에서는 Storage bucket 생성, 업로드 권한, 삭제 권한, RLS/Storage policy를 별도로 검증한다.

## 향후 처리 이력 타임라인 확장

초기 버전에서는 현재 상태와 조치/메모 1개만 관리한다. 그러나 나중에 처리 과정을 시간순으로 보여줄 수 있도록 타임라인 확장 방향을 문서에 남긴다.

향후 후보 테이블:

```text
issue_timeline_events
```

후보 필드:

- `id`
- `target_type`: `unsafe_issue` 또는 `missing_material`
- `target_id`
- `event_type`: `created`, `status_changed`, `memo_changed`, `completed`
- `status`
- `memo`
- `actor`
- `created_at`

확장 후 예시:

```text
2026.05.15 09:20 접수 - 김민수 등록
2026.05.15 10:05 조치중 - 관리자: 안전팀 전달
2026.05.15 14:20 조치중 - 관리자: 안전난간 보강 요청함
2026.05.15 16:40 완료 - 관리자: 조치 완료 확인
```

이 기능은 초기 구현 범위에는 포함하지 않는다. 다만 테이블명과 이벤트 방향을 기억해 두어 다음 확장 때 바로 이어갈 수 있게 한다.

## 화면별 요약

### 작업자 화면

- `불안전요소 등록`
  - 호선, 내용, 사진, 등록자
  - 사진 권장, 최대 3장
  - 등록 후 완료/상세 화면 표시

- `호선자재 누락`
  - 호선, 자재명, 내용, 등록자
  - 등록 후 완료/상세 화면 표시

### 관리자 화면

- 관리자 모드 ON일 때만 `관리` 메뉴 표시
- 탭: 작업자, 불안전요소, 자재누락
- 작업자 목록 추가/수정/삭제
- 기록 상태 변경
- 조치/메모 수정
- 기록 영구 삭제

## 검증 방향

구현 단계에서는 다음을 검증한다.

- 관리자 모드 OFF에서는 `관리` 메뉴가 보이지 않는다.
- 관리자 모드 ON에서는 `관리` 메뉴가 보인다.
- 작업자 목록에서 추가한 이름/팀이 두 등록 화면의 등록자 드롭다운에 표시된다.
- 삭제된 작업자는 드롭다운에서 사라지지만 기존 기록에는 이름/팀 스냅샷이 유지된다.
- 불안전요소는 사진 없이 등록 시 확인을 거친다.
- 불안전요소 사진은 최대 3장까지만 허용된다.
- 사진 업로드 후 Storage path가 `issue_photos`에 저장된다.
- 불안전요소 삭제 시 Storage 사진도 삭제된다.
- 불안전요소 목록은 상태별 그룹이며 완료 그룹은 기본 접힘이다.
- 호선자재 누락 목록은 호선별 그룹이며 완료 기록은 기본 접힘이다.
- 두 목록의 필터와 정렬이 함께 동작한다.
- 상태가 완료로 바뀌면 `completed_at`이 저장된다.
