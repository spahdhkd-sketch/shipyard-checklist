# 공기구 성격 기반 점검 필터 설계

## 배경

현재 공기구/준비물은 점검 항목마다 직접 연결하는 구조다. 이 방식은 같은 공기구를 여러 항목에 반복 연결해야 하고, 중복 점검 항목이 늘어나는 문제가 있다. 앞으로는 공기구를 개별 항목에 직접 연결하지 않고, 공기구의 성격과 점검 유형의 기준 성격을 이용해 준비물 목록과 점검 항목을 필터링한다.

## 목표

- 공기구 관리를 더보기 화면의 독립 영역으로 이동한다.
- 공기구 카드 크기를 줄여 모바일 기준 한 줄에 5개가 보이게 한다.
- 공기구마다 성격을 지정한다.
- 점검 유형마다 기준 성격을 지정한다.
- 점검 항목마다 표시 조건을 지정한다.
- 점검 전 준비물 화면에는 선택한 점검 유형에 맞는 공기구만 표시한다.
- 작업자가 선택한 공기구 성격에 따라 실제 체크리스트 항목을 표시하거나 숨긴다.
- 선택한 공기구는 기존처럼 점검 이력과 Supabase에 저장한다.
- 기존 `tool_ids` 기반의 항목별 공기구 직접 연결 UI는 제거한다.

## 공기구 성격

공기구 성격은 고정 3종이다.

- `선행`
- `후행`
- `선행/후행`

관리자는 더보기 화면의 공기구/준비물 관리에서 각 공기구의 성격을 선택한다.

## 점검 유형 기준 성격

점검 유형별 기본 기준 성격은 다음과 같다.

| 점검 유형 | 기준 성격 |
| --- | --- |
| 탑재 | 선행 |
| 선행 설치 작업 | 선행 |
| 후행 설치 작업 | 후행 |
| DP설치작업 | 후행 |
| DP검사 | 후행 |
| 압력테스트 | 선행/후행 |
| 선주선급검사 | 후행 |
| DEMO 체크리스트 | 후행 |

관리자는 섹션/항목 관리에서 점검 유형의 기준 성격을 수정할 수 있다.

## 공기구 목록 표시 규칙

점검 유형을 선택하면, 해당 점검 유형의 기준 성격에 맞는 공기구만 `사용 공기구와 준비물` 페이지에 표시한다.

- 기준 성격이 `선행`이면 `선행`, `선행/후행` 공기구를 표시한다.
- 기준 성격이 `후행`이면 `후행`, `선행/후행` 공기구를 표시한다.
- 기준 성격이 `선행/후행`이면 `선행`, `후행`, `선행/후행` 공기구를 모두 표시한다.

표시된 공기구가 없으면 준비물 페이지를 건너뛰고 체크리스트로 바로 이동한다.

## 점검 항목 표시 조건

점검 항목에는 기존 `tool_ids` 대신 표시 조건을 둔다.

- `항상 표시`
- `선행`
- `후행`
- `선행/후행`

작업자가 준비물 화면에서 공기구를 선택하면 선택된 공기구 성격을 기준으로 점검 항목을 필터링한다.

- `항상 표시` 항목은 항상 표시한다.
- `선행` 공기구를 선택하면 `선행`, `선행/후행` 조건 항목을 표시한다.
- `후행` 공기구를 선택하면 `후행`, `선행/후행` 조건 항목을 표시한다.
- `선행/후행` 공기구를 선택하면 `선행`, `후행`, `선행/후행` 조건 항목을 표시한다.

여러 공기구를 선택하면 각 공기구 성격에서 허용되는 항목 조건을 합쳐서 표시한다.

## 더보기 화면 변경

더보기 첫 화면에 `공기구/준비물 관리` 섹션을 독립적으로 추가한다. 기존처럼 특정 점검 유형의 섹션/항목 관리 안에 묶지 않는다.

공기구 카드는 모바일 기준 한 줄에 5개가 보이도록 작게 만든다.

각 공기구 카드에는 다음 정보만 우선 표시한다.

- 공기구명
- 성격 배지

관리자 모드에서는 카드 또는 행에서 다음 작업이 가능해야 한다.

- 공기구 추가
- 공기구명 수정
- 성격 수정
- 공기구 삭제

## 섹션/항목 관리 변경

섹션/항목 관리에서는 다음을 관리한다.

- 점검 유형 기준 성격
- 점검 항목 표시 조건
- 섹션 추가/수정/삭제
- 점검 항목 추가/수정/삭제

점검 항목별 공기구 직접 연결 체크박스는 제거한다.

## 중복 항목 정리

기존 항목 중 텍스트가 같은 중복 항목은 정리한다. 같은 점검 유형 안에서 텍스트가 같은 항목이 여러 개 있으면 첫 항목을 유지하고 나머지는 비활성화한다. 위험 등급, 필수 여부, 섹션은 유지되는 첫 항목 값을 따른다.

## Supabase 스키마

기존 테이블에 다음 컬럼을 추가한다.

- `public.safety_tools.nature text not null default '선행'`
- `public.safety_categories.tool_nature text not null default '선행'`
- `public.safety_items.visibility_condition text not null default '항상 표시'`

허용 값은 앱과 DB 양쪽에서 제한한다.

- `safety_tools.nature`: `선행`, `후행`, `선행/후행`
- `safety_categories.tool_nature`: `선행`, `후행`, `선행/후행`
- `safety_items.visibility_condition`: `항상 표시`, `선행`, `후행`, `선행/후행`

기존 `safety_items.tool_ids` 컬럼은 호환성을 위해 당장 삭제하지 않는다. 앱에서는 더 이상 사용하지 않고, 추후 데이터 안정화 후 제거할 수 있다.

## 저장 흐름

1. 더보기 화면에서 공기구와 성격을 관리한다.
2. 변경사항은 로컬 저장 후 Supabase `safety_tools`에 동기화한다.
3. 점검 유형 기준 성격은 `safety_categories`에 동기화한다.
4. 점검 항목 표시 조건은 `safety_items`에 동기화한다.
5. 점검 완료 시 선택한 공기구 목록은 기존처럼 `safety_inspections.tools`에 저장한다.

## 검증

- 앱 구문 검사: `node --check assets/js/app.js`
- 로컬 브라우저 스모크: `node tools/browser-smoke.mjs "<Chrome 경로>" http://127.0.0.1:4173`
- Supabase REST 검증:
  - `safety_tools`에서 `nature` 조회 가능
  - `safety_categories`에서 `tool_nature` 조회 가능
  - `safety_items`에서 `visibility_condition` 조회 가능
- 배포 후 동일 스모크를 Vercel alias URL에서 실행한다.

## 범위 제외

- 공기구 성격을 관리자가 새로 추가하는 기능은 이번 범위에 포함하지 않는다.
- 기존 `tool_ids` 컬럼 삭제는 이번 범위에 포함하지 않는다.
- 관리자 권한/RLS 보안 모델 재설계는 이번 범위에 포함하지 않는다.
